# План интеграции терминального запуска агентов в оркестратор SWARM

## Статус документа

Этот документ описывает целевую реализацию, в которой оркестратор SWARM не только рекомендует состав команды, но и сам запускает терминальных агентов через `Windows Terminal`, передаёт им стартовый промпт, контролирует число поднятых инстансов и отслеживает их регистрацию в SWARM.

Документ опирается на уже проверенную схему из [TERMINAL_AGENTS_SYSTEM_DESIGN.md](/C:/Users/Brian/Downloads/SWARM/TERMINAL_AGENTS_SYSTEM_DESIGN.md) и переводит её в план интеграции именно для оркестратора.

## Зачем нужна интеграция

Сейчас оркестратор умеет:

- анализировать проект
- рекомендовать состав команды
- ждать ручного запуска терминалов пользователем
- создавать задачи после регистрации агентов

Но запуск терминалов и `swarm join` пока остаются ручными. Это создаёт три проблемы:

- лишний ручной шаг между согласованием плана и стартом команды
- риск ошибок в именах, ролях и типах CLI
- отсутствие единого учёта, сколько терминальных агентов уже было запущено оркестратором

Целевая интеграция должна убрать этот разрыв, но сохранить контроль пользователя.

## Целевой результат

После внедрения оркестратор работает так:

1. Анализирует задачу и определяет нужный состав команды.
2. Показывает пользователю план запуска:
   - сколько агентов будет поднято
   - какие CLI будут использоваться
   - какие имена и роли будут выданы
   - какой layout будет использован
   - будет ли включён `safe` или `yolo` режим
3. Явно спрашивает подтверждение на запуск.
4. Только после подтверждения вызывает встроенный механизм запуска терминалов SWARM.
5. Каждый агент получает стартовый промпт, в котором уже зафиксированы:
   - имя
   - роль
   - CLI-тип
   - инструкция прочитать `swarm-agent` skill
   - инструкция зарегистрироваться через `swarm join`
6. Оркестратор отслеживает:
   - сколько терминалов он поднял
   - какие агенты уже зарегистрировались
   - кто не зарегистрировался или завершился с ошибкой

## Жёсткие требования

### 1. Обязательное согласование с пользователем

Оркестратор не имеет права запускать терминальные агенты без явного подтверждения пользователя.

Обязательный сценарий:

1. Оркестратор предлагает конкретный план запуска.
2. Оркестратор задаёт прямой вопрос на подтверждение.
3. Только после явного согласия выполняется запуск.

Допустимый формат:

```text
План запуска:
- 1x claude: arch-1 / architect
- 2x codex: dev-1 / developer, test-1 / tester
- 1x gemini: front-1 / developer
- layout: mixed
- режим: yolo

Подтвердите запуск 4 терминальных агентов в таком составе.
```

### 2. Оркестратор не регистрируется как агент

Это правило сохраняется без исключений.

Оркестратор:

- не выполняет `swarm join` для себя
- не получает задачи как worker
- не использует `swarm next`, `swarm lock`, `swarm done`

### 3. Рабочие агенты регистрируются сами

Оркестратор поднимает терминал и передаёт стартовый промпт, но сам `swarm join` за рабочего агента не выполняет.

Логика регистрации должна быть такой:

- агент стартует в своём CLI
- читает `swarm-agent` skill
- получает уже утверждённые имя и роль
- выполняет `swarm join --cli <type> --name <approved-name> --role <approved-role>`
- сохраняет привязку терминала к агенту согласно текущему workflow SWARM

### 4. Логика запуска не должна жить внутри LLM-инструкции

Оркестратор не должен сам собирать длинные `wt ... split-pane ... powershell ...` строки на лету.

Надёжный контракт должен быть вынесен в код SWARM:

- в PowerShell launchers
- в Python-команду `swarm terminal ...`
- в структурированный launch spec

Это важно, потому что текущий системный дизайн уже показал, что длинные inline-команды в `wt` хрупкие.

### 5. Оркестратор должен контролировать количество терминальных агентов

Нужен явный учёт:

- сколько агентов уже запущено оркестратором
- какие из них активны
- какие терминалы принадлежат какому launch session
- какие имена уже заняты

Без этого оркестратор начнёт плодить лишние терминалы и потеряет управляемость.

## Что меняется в продукте

## 1. Новый слой терминальной оркестрации

Нужно добавить отдельный слой, отвечающий за запуск и учёт терминальных агентов.

Рекомендуемые компоненты:

- `src/swarm/commands/terminal.py`
- `src/swarm/terminal/launcher_registry.py`
- `src/swarm/terminal/prompt_builder.py`
- `src/swarm/terminal/layouts.py`
- `src/swarm/terminal/preflight.py`

Назначение:

- не нагружать `agent.py` и `init.py` логикой `Windows Terminal`
- отделить бизнес-логику SWARM от логики запуска внешних CLI

## 2. Новый набор launchers

На базе текущего дизайна нужно хранить launchers как часть SWARM, а не как внешние вручную созданные файлы.

Рекомендуемая структура:

```text
src/swarm/resources/launchers/
  codex-safe-launch.ps1
  codex-yolo-launch.ps1
  claude-safe-launch.ps1
  claude-yolo-launch.ps1
  gemini-safe-launch.ps1
  gemini-yolo-launch.ps1
  opencode-safe-launch.ps1
  opencode-yolo-launch.ps1
  qwen-safe-launch.ps1
  qwen-yolo-launch.ps1
```

Каждый launcher должен:

- нормализовать `TERM`, `COLORTERM`, `TERM_PROGRAM`
- принимать `-Prompt`
- запускать нужный CLI
- использовать корректный safe/yolo профиль

## 3. Новый CLI-контракт для оркестратора

Вместо прямого вызова `wt` оркестратор должен использовать встроенные команды SWARM.

Минимально необходимый набор:

### `swarm terminal launch --spec <path>`

Запускает одну согласованную launch session по структурированному описанию.

Почему нужен `--spec`, а не длинный набор флагов:

- проще для LLM
- проще валидировать
- проще тестировать
- проще логировать и воспроизводить

### `swarm terminal status`

Показывает:

- активные launch session
- число агентов в каждой сессии
- какие агенты зарегистрированы
- какие ещё ждут регистрации

### `swarm terminal stop --session <id>`

Останавливает или помечает завершённой конкретную launch session.

### `swarm terminal reconcile --session <id>`

Сверяет:

- план запуска
- реально зарегистрированных агентов из `swarm agents`
- доступность связанных процессов

Эта команда нужна для восстановления состояния после падений или ручного закрытия окон.

## 4. Launch spec как основной интерфейс

Оркестратор должен генерировать файл launch spec и только потом вызывать `swarm terminal launch --spec ...`.

Рекомендуемый формат:

```json
{
  "version": 1,
  "working_directory": "C:\\Users\\Brian\\Downloads\\SWARM",
  "approval_mode": "yolo",
  "layout": {
    "mode": "mixed",
    "max_panes_per_window": 4
  },
  "agents": [
    {
      "cli": "claude",
      "name": "arch-1",
      "role": "architect",
      "window": 1,
      "pane": 1
    },
    {
      "cli": "codex",
      "name": "dev-1",
      "role": "developer",
      "window": 1,
      "pane": 2
    },
    {
      "cli": "gemini",
      "name": "front-1",
      "role": "developer",
      "window": 1,
      "pane": 3
    }
  ]
}
```

Важный принцип:

- имя и роль задаются здесь явно
- агент не должен придумывать их сам

## 5. Bootstrap prompt builder

Нужен отдельный генератор стартового промпта для каждого агента.

Он должен собирать промпт по шаблону и подставлять:

- `cli`
- `name`
- `role`
- путь к проекту
- путь к нужному `SKILL.md`
- правило: зарегистрироваться без дополнительных вопросов, потому что имя и роль уже согласованы с пользователем

### Пример целевого bootstrap prompt

```text
Ты рабочий агент SWARM.

Твой тип CLI: codex
Твоё имя в SWARM: dev-1
Твоя роль в SWARM: developer

Рабочая директория:
C:\Users\Brian\Downloads\SWARM

Сначала прочитай:
.codex/skills/swarm-agent/SKILL.md

Имя и роль уже согласованы с пользователем через оркестратора.
Не задавай повторных вопросов про имя и роль.

После чтения скилла зарегистрируйся:
swarm join --cli codex --name dev-1 --role developer

После регистрации выполни команду привязки терминала к агенту, которую выведет SWARM.
Затем сообщи, что регистрация завершена, и жди дальнейшей команды.
```

## Изменения в skill-системе

## 1. Изменение `swarm-agent` skill

Сейчас skill агента требует всегда спрашивать имя и роль у пользователя. Для автозапуска это слишком жёстко.

Новая редакция должна содержать исключение:

- если имя и роль явно переданы в стартовом промпте как уже согласованные пользователем через оркестратора
- агент не спрашивает их повторно
- агент использует именно эти значения в `swarm join`

Предлагаемая формулировка:

```text
Если оркестратор явно передал тебе имя и роль и указал, что они уже подтверждены пользователем, не запрашивай их повторно. Используй эти значения при регистрации через swarm join.
```

### 2. Изменение `swarm-orchestrator` skill

Нужно добавить новый workflow:

1. Сначала предложить состав команды и режим запуска.
2. Явно согласовать количество агентов и параметры запуска.
3. После согласия:
   - сформировать launch spec
   - вызвать `swarm terminal launch --spec ...`
   - дождаться регистрации агентов через `swarm agents`
4. Только после регистрации переходить к созданию задач.

Также нужно добавить запрет:

- не вызывать `wt` напрямую, если в SWARM уже есть встроенная команда `swarm terminal launch`

## Правила управления количеством агентов

Оркестратор должен управлять числом агентов по формальным правилам, а не эвристически "на глаз".

## Базовые правила

- по умолчанию не запускать больше `4` панелей в одном окне
- если нужно больше `4` агентов, использовать несколько окон
- перед запуском проверять уже активные `launch_sessions`
- не запускать новый агент с именем, которое уже занято в SWARM
- не запускать новый агент с тем же именем внутри незавершённой launch session

## Рекомендуемые лимиты первой версии

- максимум `4` панели в одном окне
- максимум `8` агентов в одном действии запуска без отдельного подтверждения пользователя
- если запрошено больше `8`, оркестратор обязан предупредить о сложности мониторинга и предложить поэтапный запуск

## Политика повторного использования

Перед новым запуском оркестратор должен проверять:

- есть ли уже свободные зарегистрированные агенты с подходящим CLI и ролью
- нужно ли запускать новых, или можно переиспользовать существующих

Это предотвратит бесконтрольный рост числа терминалов.

## Пользовательский workflow

## Сценарий 1. Обычный запуск

1. Пользователь просит оркестратора спланировать работу.
2. Оркестратор анализирует проект.
3. Оркестратор предлагает:
   - состав команды
   - layout
   - safe/yolo режим
   - количество терминалов
4. Пользователь подтверждает запуск.
5. Оркестратор создаёт launch spec.
6. Оркестратор вызывает `swarm terminal launch --spec ...`.
7. Поднятые агенты читают skill и сами выполняют `swarm join`.
8. Оркестратор проверяет `swarm agents`.
9. После регистрации создаёт задачи.

## Сценарий 2. Частичный запуск

Если пользователь подтверждает не весь план, а только часть:

- оркестратор запускает только подтверждённых агентов
- пересчитывает план задач под фактический состав команды
- явно сообщает, какие роли временно не покрыты

## Сценарий 3. Повторный запуск после неудачи

Если часть агентов не зарегистрировалась:

- оркестратор выполняет `swarm terminal status`
- затем `swarm terminal reconcile`
- показывает пользователю, кто не поднялся
- спрашивает, нужно ли перезапустить только проблемные терминалы

## Сценарий 4. Масштабирование команды

Если в процессе работы понадобились ещё агенты:

- оркестратор сначала объясняет, зачем нужно расширение
- показывает новый состав
- отдельно согласует дополнительный запуск
- запускает только добавочных агентов

## Реализация layout-режимов

На первом этапе достаточно поддержать три режима.

## `single`

Один агент в одном окне.

Используется для:

- точечного добавления одного нового исполнителя
- безопасного старта без сложной раскладки

## `mixed`

Несколько разных CLI в одном окне `Windows Terminal`.

Используется для:

- компактного старта типовой команды до `4` агентов

## `multi-window`

Несколько окон, в каждом по одному или нескольким агентам.

Используется для:

- команд больше `4` агентов
- ситуаций, где нужен разнос по экранам и более читаемый мониторинг

Важно:

- `grid4` как внутренний layout может использоваться реализацией
- но для пользователя лучше оставить понятные режимы `single`, `mixed`, `multi-window`

## Preflight-check перед запуском

Перед `swarm terminal launch` должна выполняться проверка:

- существует ли `wt.exe`
- существует ли рабочая директория
- существуют ли launchers для нужных CLI и режима
- существует ли сам исполняемый CLI
- нет ли конфликта по именам агентов
- не превышены ли лимиты количества агентов

### Отдельное правило для OpenCode

Если не найден `opencode.ps1` или shim повреждён:

- запуск этого агента помечается как невозможный
- пользователю предлагается repair flow
- в документации и ошибке используется проверенный вариант:

```powershell
npm install -g opencode-ai
```

Результат этапа:

- пользователь понимает новый workflow и точки подтверждения запуска

## Этап 6. Тестирование

Нужно покрыть минимум:

- валидацию launch spec
- генерацию bootstrap prompt

- защиту от дублирующихся имён
- корректный выбор launcher для `safe` и `yolo`
- интеграционный happy path для `swarm terminal launch`
- обновлённые шаблоны `swarm init`

## Критерии готовности

Фича считается готовой, когда выполняются все условия:

1. Оркестратор всегда сначала показывает пользователю состав запуска и спрашивает подтверждение.
2. После подтверждения может запустить минимум:
   - одного агента
   - mixed-команду до `4` агентов
3. Каждый поднятый агент получает стартовый промпт с уже заданными именем и ролью.
4. Агент после старта способен сам зарегистрироваться через `swarm join`.
5. Оркестратор может увидеть, кто зарегистрировался, а кто нет.
6. Оркестратор не запускает лишних агентов сверх согласованного количества.
7. В логах видно жизненный цикл launch session.
8. `swarm init` создаёт обновлённые `SKILL.md` с новой логикой.

## Основные риски

### 1. Слишком умный bootstrap prompt

Если стартовый промпт будет длинным и расплывчатым, разные CLI начнут интерпретировать его по-разному.

Вывод:
