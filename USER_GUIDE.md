# SWARM — Руководство пользователя

## Введение

SWARM — это система для координации нескольких LLM-агентов (Claude, Codex, Gemini, OpenCode, Qwen), работающих параллельно над одной кодовой базой.

### Два режима работы

SWARM поддерживает два режима — выбирайте тот, который вам удобнее:

| | Ручной режим | Режим с оркестратором |
|---|---|---|
| **Кто планирует задачи?** | Вы сами | LLM-оркестратор |
| **Кто создаёт задачи?** | Вы через `swarm task add` | Оркестратор через `swarm task add` |
| **Кто подбирает команду?** | Вы решаете, каких агентов запустить | Оркестратор рекомендует состав |
| **Кто проверяет результат?** | Вы сами | Оркестратор делает ревью, вы утверждаете |
| **Когда использовать?** | Вы точно знаете, какие задачи нужны | Нужна помощь с декомпозицией и планированием |

В ручном режиме вы выступаете и **Лидером**, и **оркестратором** — полный контроль в ваших руках. В режиме с оркестратором вы делегируете планирование LLM-агенту, а сами остаётесь Лидером: утверждаете план, запускаете агентов, контролируете процесс.

Оба режима можно комбинировать: начать с оркестратором, а потом добавлять задачи вручную по ходу работы.

## Система скиллов (SKILL.md)

При выполнении `swarm init` в проекте создаются файлы `SKILL.md` — инструкции, которые LLM-агенты читают для понимания своей роли в SWARM. Скиллы создаются для каждого поддерживаемого CLI:

```
.claude/skills/swarm-agent/SKILL.md
.claude/skills/swarm-orchestrator/SKILL.md
.codex/skills/swarm-agent/SKILL.md
.codex/skills/swarm-orchestrator/SKILL.md
.gemini/skills/swarm-agent/SKILL.md
... и т.д.
```

### Скилл агента (`swarm-agent`)

Определяет поведение LLM как **рабочего агента**. Когда вы говорите агенту "прочитай SKILL.md и зарегистрируйся", он находит этот файл и следует инструкциям.

**За что отвечает:**
- Регистрация — агент спрашивает у пользователя имя и роль, выполняет `swarm join`
- Рабочий цикл — получение задачи → блокировка файла → правки → разблокировка → завершение
- Правила блокировок — одна блокировка за раз, разблокировка сразу после правок
- Heartbeat — периодическое обновление при долгих операциях
- Формат резюме — что писать в `--summary` при завершении задачи
- Ограничения — агент знает, что ему **запрещено** управлять задачами, снимать чужие блокировки и использовать `--help`

### Скилл оркестратора (`swarm-orchestrator`)

Определяет поведение LLM как **оркестратора**. Оркестратор **не** регистрируется как агент — он управляет процессом.

**За что отвечает:**
- Профили CLI — знание сильных сторон каждого CLI (Claude для архитектуры, Codex для алгоритмов, Gemini для фронтенда и т.д.)
- Рекомендация команды — оркестратор предлагает, какие CLI запустить и с какими ролями
- Декомпозиция задач — разбиение цели на атомарные задачи с приоритетами и зависимостями
- Создание задач — оркестратор сам выполняет `swarm task add` после утверждения плана
- Мониторинг — отслеживание прогресса, реагирование на зависание агентов
- Ревью — проверка качества результата, создание задач на доработку
- Ограничения — оркестратор знает, что ему **запрещено** выполнять `swarm join`, `swarm next`, `swarm lock` и другие команды агента

### Как агенты находят скиллы

Каждый CLI имеет свой механизм загрузки скиллов:
- **Claude** — читает `.claude/skills/*/SKILL.md` автоматически
- **Codex** — читает `.codex/skills/*/SKILL.md`
- **Gemini** — читает `.gemini/skills/*/SKILL.md`
- **OpenCode** — читает `.opencode/skills/*/SKILL.md`
- **Qwen** — читает `.qwen/skills/*/SKILL.md`

Содержимое скиллов одинаковое для всех CLI (адаптировано только имя CLI-типа в примерах).

## Работа с оркестратором

Оркестратор — это LLM-агент (рекомендуется Claude), которому вы делегируете планирование, распределение задач и контроль качества. Он работает в отдельном терминале и управляет SWARM через CLI-команды.

### Пошаговый workflow

#### Шаг 1. Инициализация

```bash
cd ваш-проект
swarm init
```

#### Шаг 2. Запуск оркестратора

Откройте терминал с LLM и скажите:

> "Ты оркестратор SWARM. Прочитай скилл оркестратора и спланируй работу над \<ваша цель\>."

Оркестратор:
1. Прочитает `SKILL.md` оркестратора
2. Изучит кодовую базу (прочитает файлы, поищет паттерны)
3. Декомпозирует цель на атомарные задачи
4. Предложит вам состав команды:

```
Для этого проекта рекомендую запустить:

Терминал 1: claude  → роль: architect,  имя: arch-1
Терминал 2: codex   → роль: developer,  имя: dev-1
Терминал 3: gemini  → роль: developer,  имя: front-1
Терминал 4: codex   → роль: tester,     имя: test-1

В каждом терминале выполните:
  swarm join --cli <тип> --name <имя> --role <роль>
```

5. Предложит план задач с приоритетами и зависимостями

**Важно:** утвердите план перед тем, как оркестратор начнёт создавать задачи. Вы можете скорректировать состав команды и набор задач.

#### Шаг 3. Запуск агентов

Откройте отдельный терминал для каждого агента из рекомендации. В каждом терминале запустите CLI и скажите:

> "Прочитай SKILL.md агента и зарегистрируйся через swarm join"

Агент спросит имя и роль (используйте те, что рекомендовал оркестратор), затем зарегистрируется:

```bash
swarm join --cli codex --name dev-1 --role developer
```

#### Шаг 4. Оркестратор создаёт задачи

Когда все агенты зарегистрированы, скажите оркестратору:

> "Агенты зарегистрированы. Проверь через `swarm agents` и создавай задачи."

Оркестратор проверит, что все на месте, и создаст задачи:

```bash
swarm task add --desc "Спроектировать схему БД" --priority 1 --name arch-1
swarm task add --desc "Реализовать REST API" --priority 2 --role developer --cli codex --depends-on 1
swarm task add --desc "Создать UI формы" --priority 2 --cli gemini --depends-on 1
swarm task add --desc "Написать тесты для API" --priority 3 --role tester --depends-on 2
```

#### Шаг 5. Запуск работы агентов

Скажите каждому агенту в его терминале:

> "Начинай работать. Выполни `swarm next --agent твоё-имя` чтобы получить задачу."

Или оркестратор может выполнить `swarm start --all`, чтобы сигнализировать агентам о начале.

#### Шаг 6. Мониторинг

Вы можете следить за прогрессом двумя способами:

1. **Дашборд** — в отдельном терминале:
   ```bash
   swarm monitor    # или swarm tui
   ```

2. **Через оркестратора** — попросите его проверить прогресс:
   > "Как дела? Проверь статус."

   Оркестратор выполнит `swarm task list` и `swarm logs` и даст отчёт.

Если агент завис — оркестратор может:
- Закрыть задачу: `swarm task close <ID> --reason "Завис"`
- Снять блокировки: `swarm unlock --force --file X`
- Пересоздать задачу для другого агента

#### Шаг 7. Ревью результата

Когда все задачи завершены, скажите оркестратору:

> "Все задачи выполнены. Сделай ревью."

Оркестратор:
1. Прочитает резюме каждой задачи (`swarm task list --all`)
2. Изучит изменённые файлы
3. Оценит качество: код соответствует описанию? нет конфликтов? тесты есть?
4. При необходимости — создаст новые задачи на доработку

#### Шаг 8. Финализация

Оркестратор подведёт итоги:
- Что было сделано
- Какие файлы затронуты
- Что требует внимания
- Предложит запустить тесты

### Профили CLI-агентов

Оркестратор учитывает сильные стороны каждого CLI при распределении задач:

| CLI | Сильные стороны | Лучшие роли |
|-----|----------------|-------------|
| `claude` | Архитектура, системное проектирование, сложная разработка | `architect`, `developer` |
| `codex` | Глубокий кодинг, поиск багов, код-ревью, алгоритмы | `developer`, `tester` |
| `gemini` | Фронтенд, UI-компоненты, вёрстка, стили | `developer` (frontend) |
| `opencode` | Документация, рутинные задачи, правка по описанию | `developer`, `devops` |
| `qwen` | Общая разработка, скрипты, утилиты | `developer` |

### Примеры составов команды

**Фуллстек-разработка:**
- 1x claude (architect) — проектирование и сложная логика
- 1x gemini (developer) — фронтенд/UI
- 1x codex (developer) — бэкенд и алгоритмы
- 1x codex (tester) — тесты и ревью

**Бэкенд/API:**
- 1x claude (architect) — архитектура
- 2x codex (developer) — реализация
- 1x opencode (tester) — тесты по описанию

**Фронтенд:**
- 1x claude (architect) — архитектура компонентов
- 2x gemini (developer) — UI-компоненты
- 1x codex (tester) — тесты

**Баг-фикс / рефакторинг:**
- 2x codex (developer) — поиск и исправление
- 1x opencode (developer) — рутинные правки

### Фазы работы оркестратора (справочник)

| Фаза | Что делает оркестратор | Что делаете вы |
|------|----------------------|----------------|
| 1. Анализ | Изучает кодовую базу, декомпозирует цель | Ставите цель, отвечаете на вопросы |
| 2. Подбор команды | Рекомендует CLI + роли + имена | Утверждаете или корректируете |
| 3. Ожидание агентов | Проверяет `swarm agents` | Запускаете агентов в терминалах |
| 4. Создание задач | `swarm task add` с приоритетами и зависимостями | Утверждаете план задач |
| 5. Мониторинг | Следит за прогрессом, решает проблемы | Наблюдаете через `swarm monitor` |
| 6. Ревью | Проверяет результат, создаёт fix-задачи | Утверждаете результат |

## Ручной режим (Лидер без оркестратора)

Если вы предпочитаете полный контроль — вы сами выступаете Лидером и оркестратором. Вы сами планируете задачи, подбираете агентов и проверяете результат.

### Быстрый старт

#### 1. Установка

```bash
# Из директории SWARM
pip install -e .
```

#### 2. Инициализация проекта

Перейдите в папку вашего проекта и выполните:

```bash
cd ваш-проект
swarm init
```

Это создаст:
- `swarm.db` — база данных для задач и агентов
- `SKILL.md` — инструкции для LLM-агентов (в папках `.claude/`, `.codex/`, `.gemini/` и др.)

#### 3. Создание задач

```bash
# Простая задача
swarm task add --desc "Реализовать авторизацию" --priority 1

# Задача для конкретной роли
swarm task add --desc "Спроектировать API" --priority 1 --role architect

# Задача с зависимостью (выполнится после задачи #1)
swarm task add --desc "Написать тесты" --priority 2 --depends-on 1
```

**Приоритеты:** 1 (наивысший) — 5 (наименьший)

**Роли:** `architect`, `developer`, `tester`, `devops`

**Типы CLI:** `claude`, `codex`, `gemini`, `opencode`, `qwen`

#### 4. Запуск агентов

Откройте новый терминал для каждого агента:

```bash
# Терминал 1: Запустите Claude CLI
claude

# В Claude скажите:
# "Зарегистрируйся через swarm join"
```

Агент выполнит:
```bash
swarm join
# Введёт: тип CLI, имя, роль
```

**⚠ ВАЖНО: Агенты должны использовать параметр `--agent`**

После регистрации агент должен **запомнить своё имя** и использовать его во всех командах:

```bash
# Регистрация
swarm join --cli codex --name worker1 --role developer

# Все последующие команды — с --agent
swarm next --agent worker1
swarm lock файл.py --agent worker1
swarm unlock --file файл.py --agent worker1
swarm heartbeat --agent worker1 --quiet
swarm done --summary "..." --agent worker1
```

Это позволяет запускать несколько агентов одного типа (например, 5 Codex) без конфликтов.

Повторите для каждого агента в отдельном терминале.

#### 5. Запуск монитора

В отдельном терминале:

```bash
swarm monitor
```

Вы увидите live-дашборд с 4 панелями:
- **Агенты** — статус каждого агента
- **Задачи** — очередь задач
- **Блокировки** — какие файлы заблокированы
- **Активность** — последние события

#### 6. Начало работы

**В каждом терминале агента** скажите ему текстом:

> "Начинай работать. Выполни `swarm next --agent твоё-имя` чтобы получить задачу."

Агент выполнит цикл:
1. Получит задачу (`swarm next --agent имя`)
2. Заблокирует **один** файл (`swarm lock файл --agent имя`)
3. Внесёт правки и сразу разблокирует файл (`swarm unlock --file файл --agent имя`)
4. При долгой работе обновит heartbeat (`swarm heartbeat --agent имя --quiet`)
5. Завершит задачу (`swarm done --agent имя`)
6. Возьмёт следующую задачу

**Важно:** Агенты — это LLM в отдельных терминалах. Они не "слушают" базу данных автоматически. Вы должны **вручную** сказать каждому агенту начать работу.

### Типичная сессия

1. **Подготовка**
   ```bash
   cd my-project
   swarm init
   ```

2. **Создание задач**
   ```bash
   swarm task add --desc "Спроектировать архитектуру" --priority 1 --role architect
   swarm task add --desc "Реализовать UserController" --priority 2 --role developer --depends-on 1
   swarm task add --desc "Написать тесты" --priority 3 --role tester --depends-on 2
   ```

3. **Запуск агентов** (в отдельных терминалах)
   - Терминал 1: Claude как architect
   - Терминал 2: Claude как developer
   - Терминал 3: Claude как tester

4. **Мониторинг**
   ```bash
   swarm monitor
   ```

5. **Добавление задач по ходу работы**
   ```bash
   swarm task add --desc "Исправить баг в API" --priority 1 --role developer
   ```

## Команды Лидера

### Управление задачами

```bash
# Создать задачу
swarm task add --desc "Описание" --priority 1 [--role РОЛЬ] [--cli ТИП] [--name ИМЯ] [--depends-on ID]

# Список задач
swarm task list [--status pending|in_progress|done] [--done]

# Принудительно закрыть задачу (если агент завис)
swarm task close <ID> [--reason "Причина"]

# Назначить задачу конкретному агенту
swarm task assign <ID> --agent <имя>

# Примеры
swarm task add --desc "Рефакторинг базы данных" --priority 2 --role developer
swarm task add --desc "Ревью кода" --priority 3 --name alice
swarm task list --status pending
swarm task assign 5 --agent bob     # Назначить задачу #5 агенту bob
swarm task close 5 --reason "Агент завис"
```

### Управление агентами

```bash
# Список агентов
swarm agents

# Удалить неактивных агентов (мёртвые процессы + >30 мин без heartbeat)
swarm agents --cleanup

# Удалить ВСЕХ агентов (полная очистка)
swarm agents --cleanup --force
```

### Мониторинг

```bash
# Live-дашборд (Ctrl+C для выхода)
swarm monitor

# С полным описанием задач
swarm monitor --full

# TUI-монитор со скроллингом (рекомендуется)
swarm tui

# Настроить интервал обновления (по умолчанию 2 сек)
swarm monitor --refresh 5

# Список задач с полным описанием и резюме
swarm task list            # Активные задачи
swarm task list --all      # Все задачи (включая завершённые)

# Журнал событий
swarm logs
swarm logs --limit 100         # Последние 100 событий
swarm logs --agent alice       # События агента alice
swarm logs --task 5            # События задачи #5
```

**Столбцы в мониторе задач:**
| Столбец | Описание |
|---------|----------|
| ID | Номер задачи |
| P | Приоритет (1-5) |
| Статус | pending/in_progress/done/blocked |
| Роль | Для какой роли задача (`--role`) |
| После | Зависимость (`--depends-on`) |
| Назначен | Кому назначена (`--name` или `task assign`) |
| Работает | Кто сейчас выполняет |

### Блокировки

```bash
# Принудительно снять блокировку (если агент завис)
swarm unlock --force --file src/main.py

# Снять все блокировки
swarm unlock --all --force
```

Принудительная разблокировка доступна только Лидеру или оркестратору. Агент может снимать только свои обычные блокировки без `--force`.

### Фильтрация задач

| Параметр | Описание | Пример |
|----------|----------|--------|
| `--role` | Только агенты с этой ролью | `--role architect` |
| `--cli` | Только агенты этого типа | `--cli claude` |
| `--name` | Только агент с этим именем | `--name alice` |
| `--depends-on` | Выполнить после задачи | `--depends-on 5` |

### Зависимости задач

```bash
# Задача #1: архитектура
swarm task add --desc "Спроектировать" --priority 1

# Задача #2: реализация (после #1)
swarm task add --desc "Реализовать" --priority 2 --depends-on 1

# Задача #3: тесты (после #2)
swarm task add --desc "Тесты" --priority 3 --depends-on 2
```

Задача #2 не будет доступна, пока #1 не завершена.

## Решение проблем

### Агент завис / не отвечает

1. Проверьте heartbeat в `swarm agents`
2. Если heartbeat старый, сначала проверьте: процесс действительно мёртв или агент просто долго думает/редактирует локально
3. Попросите агента выполнить `swarm heartbeat --agent <имя> --quiet`, если он жив и продолжает работу
4. Если агент действительно завис, **закройте его задачу принудительно:**
   ```bash
   swarm task close <ID> --reason "Агент завис"
   ```
   Это автоматически снимет блокировки и освободит агента.

5. Или снимите блокировки вручную:
   ```bash
   swarm unlock --force --file заблокированный_файл.py
   swarm unlock --all --force   # Все блокировки
   ```

6. Удалите мёртвых агентов:
   ```bash
   swarm agents --cleanup
   ```

### Задача не назначается

Проверьте фильтры:
- Есть ли агент с нужной ролью?
- Завершена ли зависимость (`depends_on`)?

```bash
swarm task list    # Проверить статус задач
swarm agents       # Проверить роли агентов
```

### Конфликт файлов

Система предотвращает конфликты через блокировки. Если агент ждёт файл:
- Это нормально — он подождёт освобождения
- В мониторе видно, кто блокирует файл

### Очистка

```bash
# Удалить неактивных агентов (по PID и heartbeat)
swarm agents --cleanup

# Удалить ВСЕХ агентов
swarm agents --cleanup --force

# Переинициализировать (осторожно — удалит все данные)
```

## Лучшие практики

1. **Создавайте все задачи до запуска агентов** — так приоритеты будут корректными

2. **Используйте роли** — architect для проектирования, developer для кода, tester для тестов

3. **Пишите чёткие описания** — агенты это LLM, им нужны ясные инструкции

4. **Следите за блокировками** — агент должен держать только один lock одновременно и снимать его сразу после завершения правок в файле

5. **Используйте зависимости** — для последовательных задач

6. **Держите монитор открытым** — так вы видите прогресс в реальном времени
7. **Не давайте агентам функции оркестратора** — `task add`, `task assign`, `task close` выполняют только пользователь или оркестратор
8. **Не считайте старый heartbeat автоматическим зависанием** — сначала отличайте устаревший heartbeat от реально мёртвого процесса
9. **Используйте оркестратора для сложных проектов** — декомпозиция и подбор команды экономят время
10. **Комбинируйте режимы** — запустите оркестратора для планирования, а потом добавляйте задачи вручную по ходу работы

## Дашборд монитора

Есть два варианта:
- `swarm monitor` — простой Rich-дашборд
- `swarm tui` — TUI со скроллингом (рекомендуется для большого числа задач)

```
┌─────────────────────────────┬───────────────────────────────────────────────┐
│       ПАНЕЛЬ АГЕНТОВ        │              ПАНЕЛЬ ЗАДАЧ                     │
│                             │                                               │
│  #1 claude/alice/architect  │  ID  P  Статус    Роль  После Назнач. Работает│
│     Статус: WORKING [#7]    │  #7  1  🔄 in_prog dev    -    alice   alice  │
│     Heartbeat: 12с          │  #8  2  ⏳ pending test   #7   bob      -     │
│                             │  #9  3  ⏳ pending  -     #8    -       -     │
│  #2 codex/bob/developer     │                                               │
│     Статус: IDLE            │                                               │
├─────────────────────────────┼───────────────────────────────────────────────┤
│    ПАНЕЛЬ БЛОКИРОВОК        │    ПАНЕЛЬ АКТИВНОСТИ                          │
│                             │                                               │
│  src/api.py -> alice [12м]  │  14:32 ▶ #7 alice  Task started               │
│  src/db.py  -> alice [12м]  │  14:31 ✓ #6 bob    Task done                  │
│                             │  14:30 🔒    alice  Lock api.py               │
└─────────────────────────────┴───────────────────────────────────────────────┘
```

**Панель задач показывает:**
- **Роль** — для какой роли задача (`--role architect`)
- **После** — зависимость, после какой задачи (`--depends-on 5`)
- **Назначен** — кому назначена задача (`--name` или `task assign`)
- **Работает** — кто сейчас выполняет

**Панель активности показывает:**
- Номер задачи (`#7`) для событий, связанных с задачами
- Имя агента, выполнившего действие
- Тип события (▶ старт, ✓ готово, 🔒 блокировка)

**Управление в TUI (`swarm tui`):**
- **↑↓** — скроллинг списков
- **R** — обновить данные
- **D** — показать/скрыть завершённые
- **Q** — выход

**Цвета статусов:**
- 🟢 Зелёный — working (работает)
- 🟡 Жёлтый — waiting (ждёт блокировку)
- ⚪ Серый — idle (ожидает задачу)
- 🔴 Красный — heartbeat >5 мин (возможно завис)

## Справка по командам

Встроенная CLI-справка в SWARM отключена намеренно.

- Агентам нельзя использовать `swarm --help` и аналогичные вызовы для просмотра полного списка команд.
- Рабочие команды для агента должны приходить из соответствующего `SKILL.md`.
- Команды оркестратора и лидера также должны документироваться в инструкциях, а не через встроенный help.

## Краткая шпаргалка

### Для Лидера (ручной режим)

| Действие | Команда |
|----------|---------|
| Создать задачу | `swarm task add --desc "..." --priority 1` |
| Список задач | `swarm task list` |
| Список всех задач | `swarm task list --all` |
| **Назначить задачу** | `swarm task assign <ID> --agent <имя>` |
| **Закрыть задачу** | `swarm task close <ID>` |
| Список агентов | `swarm agents` |
| Удалить мёртвых | `swarm agents --cleanup` |
| Удалить всех | `swarm agents --cleanup --force` |
| Мониторинг | `swarm monitor` |
| **TUI со скроллингом** | `swarm tui` |
| Монитор + описание | `swarm monitor --full` |
| Журнал | `swarm logs` |
| Снять блокировку | `swarm unlock --force --file X` |
| Снять все блокировки | `swarm unlock --all --force` |

### Для работы с оркестратором

| Шаг | Что сказать оркестратору |
|-----|--------------------------|
| Запуск | "Ты оркестратор SWARM. Прочитай скилл оркестратора и спланируй работу над \<цель\>." |
| После анализа | Утвердите план или скорректируйте |
| После регистрации агентов | "Агенты зарегистрированы. Проверь `swarm agents` и создавай задачи." |
| Проверка прогресса | "Как дела? Проверь статус." |
| После завершения | "Все задачи выполнены. Сделай ревью." |
| Доработка | "Создай задачу на исправление \<проблема\>." |
